## 1. 背景与动机

### 1.1 问题陈述

在TTRPG（桌面角色扮演游戏）AI化的浪潮中，D&D 5e已拥有相对丰富的开源工具生态（d20、5etools、Avrae等），但COC（Call of Cthulhu，《克苏鲁的呼唤》）7e的开源工具处于极度贫瘠状态。截至2026年5月，GitHub上与COC直接相关的开源仓库仅约18个，且呈现"低星标、低维护、低完成度"的三低特征。

与此同时，COC与D&D在底层规则哲学上存在根本性差异：
- D&D复杂性集中在战斗（动作经济、法术位、覆盖/遮蔽）
- COC复杂性集中在调查逻辑、线索关联和SAN（理智）的叙事性衰退

这意味着直接将D&D的AI DM架构移植到COC场景将导致灾难性的体验崩塌。我们需要从零开始构建专门用于COC的AI守秘人（Keeper）系统。

### 1.2 市场机遇

COC工具生态的结构性缺失既是挑战，也是构建差异化系统的战略机遇：
1. **版权壁垒**：混沌元素（Chaosium）的COC 7e规则未发布类似D&D SRD的开放授权文档，开发者无法合法地构建和分发包含完整规则数据的开源工具。
2. **市场体量差异**：D&D占据TTRPG市场约60%以上的份额，其玩家基数和开发者社区远超COC。
3. **系统复杂性方向不同**：D&D的复杂性是"数值密集型"，相对容易形式化为代码；COC的复杂性是"语义密集型"（调查逻辑链、理智衰减曲线、恐怖氛围营造），这对传统程序化建模构成挑战——但也恰好是LLM（大语言模型）的技术优势所在。

### 1.3 项目目标

本阶段的目标是完成技术调研、可行性分析和深度研究报告，为后续开发提供坚实的理论基础和技术路线。

## 2. 目标与范围

### 2.1 核心目标

1. **完成COC与D&D在AI DM/KP系统中的架构差异分析**
2. **评估各LLM模型在COC场景下的表现**（氛围文风控制、长程逻辑一致性、信息节流与悬念控制等）
3. **设计系统提示词工程框架**（恐怖氛围的编码化约束、渐进式紧张度曲线、强制掷骰前置协议等）
4. **定义线索知识图谱架构**（线索实体Schema、线索生命周期管理、线索关联图）
5. **设计SAN状态机模型**（多维度的心理状态系统、理智损失历史、恐惧症与躁狂症建模）

### 2.2 范围边界

**包含**：
- COC 7e核心规则的深度分析
- LLM模型评估与选型建议
- 系统提示词框架设计
- 线索知识图谱与SAN状态机的数据模型设计
- 技术栈选型与架构设计

**不包含**：
- 实际代码实现
- 用户界面设计
- 部署方案设计

## 4. 功能需求

### 4.1 市场生态与COC工具现状分析

**描述**：分析COC工具生态的现状，识别结构性缺失的根本原因，并盘点现有可借力的资源。

**子需求**：
1. 分析COC工具生态的现状（骰子机器人、角色生成器、战斗模拟、追逐追踪、规则引擎参考）
2. 对比COC与D&D的开源工具生态差异
3. 分析COC工具稀缺的根本原因（版权壁垒、市场体量差异、系统复杂性方向不同）
4. 盘点现有可借力的资源（sealdice-core/ext_coc7.go、Chaosium官方资源、社区内容生态、通用TRPG框架）

### 4.2 核心LLM在COC场景下的能力评估

**描述**：评估各LLM模型在COC场景下的表现，确定最适合的模型作为AI KP的叙事引擎。

**评估维度**（按重要性排序）：
1. **氛围文风控制**（最关键）：能否持续输出洛夫克拉夫特式的"宇宙恐怖"文本
2. **长程逻辑一致性**：能否在数十个回合中维持NPC的谎言一致性、线索的时间线自洽
3. **信息节流与悬念控制**：能否抵抗"一次性倾倒所有线索"的冲动
4. **概率性判定直觉**：对d100系统的"感觉"
5. **SAN机制的叙事理解**：理解不同层级的理智损失对应不同的心理反应
6. **角色分离与信息隐藏**：严格维护信息不对称

**评估模型**：
- Claude系列（Anthropic）
- GPT-4o（OpenAI）
- Gemini 2.5 Pro（Google）
- 开源模型（Llama 4、Qwen3等）
- DeepSeek系列

### 4.3 系统提示词工程框架设计

**描述**：设计构建克苏鲁守秘人的"职业契约"，将恐怖氛围控制、信息释放节奏、SAN叙事衰减等非线性需求编码化为系统提示词。

**核心模块**：
1. **恐怖氛围的编码化约束**
   - 禁止"安全叙事"（Safety Narrative Prohibition）
   - "展示而非告知"的恐怖编码（Show, Don't Tell Horror）
   - 渐进式紧张度曲线（Escalation Curve）

2. **强制掷骰前置与COC特有的判定协议**
   - 技能检定的触发规则（无需掷骰 vs 必须掷骰）
   - 奖励骰与惩罚骰的叙事锚定
   - SAN检定的叙事区分

3. **信息控制与线索释放协议**
   - 每个NPC的信息边界定义（known_facts vs unknown_facts）
   - 线索释放的节奏控制（三线索法则、反"信息倾倒"机制、错误推论的尊重）

4. **出戏干预（OOC）与玩家能动性保护**
   - 事实核查指令
   - 基调调整指令
   - 节奏调整指令

### 4.4 结构化状态追踪：COC的调查与理智双状态机

**描述**：设计COC特有的状态追踪模型，以"调查状态图"和"理智状态机"为双核心。

**数据模型设计**：

1. **调查状态图（线索作为一等公民）**
   - 线索实体JSON Schema设计（`clue_id`, `type`, `description_for_players`, `description_for_keeper`, `connections`, `status`, `investigators_who_know`, 等）
   - 线索生命周期：`hidden` → `found` → `found_but_not_understood` → `understood` → `connected_to_main_mystery` → `resolved`
   - 线索关联图维护

2. **理智（SAN）状态机的精细化建模**
   - 多维度的心理状态系统（`current_san`, `temporary_insanity`, `indefinite_insanity`, `phobias`, `sanity_loss_history`, `current_mental_state`）
   - 理智损失历史追踪
   - 恐惧症与躁狂症数组

3. **COC角色卡数据结构**
   - 属性（STR, CON, SIZ, DEX, APP, INT, POW, EDU）
   - 衍生值（HP, SAN, Luck, MP, Move, Damage_Bonus, Build, Dodge）
   - 技能（扁平的技能横向分布）
   - 物品、伤口、联系人

4. **追逐（Chase）系统的规则建模**
   - 追逐状态对象（`active`, `participants`, `distance_between`, `hazards_deployed`, `round`, `escape_condition`, `capture_condition`）

### 4.5 长程战役记忆机制：线索网络的持久化与检索

**描述**：设计线索知识图谱架构，解决COC特有的"线索时间分散性"挑战。

**核心设计**：
1. **线索知识图谱架构**
   - 线索以节点（Node）形式存储，关系的类型决定了AI如何基于当前发现触发"灵感时刻"的叙事
   - 图谱与LLM的交互协议（识别玩家当前行动指向的线索节点、1-2跳范围检索关联节点、将关联节点注入LLM的提示词）

2. **信息不对称的持久化管理**
   - 维护"当前暴露度"——衡量调查员距离真相有多近的数值指标
   - AI在生成每个回应前，被明确告知当前暴露度级别，并被约束只能在对应级别下释放信息

3. **线索图谱的持久化与Obsidian集成**
   - 每条线索的Markdown笔记天然包含双向链接（`[[线索ID]]`）
   - 线索关系图不仅可被AI查询，也可被人类KP直接阅读和修改

## 5. 非功能需求

### 5.1 研究报告质量

1. **深度**：研究报告应深入剖析COC AI KP系统的核心技术挑战，而非浅尝辄止
2. **广度**：研究报告应覆盖市场生态、LLM评估、提示词工程、状态追踪、记忆机制、幻觉抑制、多模态资产生成等各个维度
3. **可行性**：研究报告提出的技术架构和实施路径应是切实可行的，而非空中楼阁

### 5.2 文档可读性

1. **结构清晰**：研究报告应有清晰的章节结构和目录导航
2. **示例丰富**：应有大量的"错误模式 vs 正确模式"对比示例
3. **图表辅助**：应有表格、流程图等图表辅助理解（如COC与D&D的对比表、紧张度曲线表等）

### 6.1 交付物验收

1. ✅ **深度研究报告文档完成**
   - 字数 ≥ 15000 字
   - 章节完整（市场生态分析、LLM能力评估、提示词工程、状态追踪、记忆机制、幻觉抑制、多模态资产生成、实施蓝图）

   - 前端：TypeScript + React + Vite
   - 后端：Express + Socket.IO
   - 数据库：SQLite (sql.js)
   - LLM：DeepSeek API（OpenAI兼容接口）

3. ✅ **项目仓库初始化**
   - 创建 `coc-ai-keeper/` 目录
   - 初始化 `package.json`
   - 配置TypeScript、Vite、Vitest

### 6.2 质量验收

1. ✅ **研究报告通过团队评审**
   - 无事实性错误
   - 技术架构设计合理
   - 实施路径可行

2. ✅ **LLM评估结论明确**
   - 明确推荐DeepSeek作为主力模型
   - 明确说明选择理由（性价比高、OpenAI兼容接口、中文支持好）
