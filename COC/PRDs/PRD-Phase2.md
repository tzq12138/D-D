## 1. 背景与动机

### 1.1 从规则引擎到可玩应用

Phase 1完成了基础架构和规则引擎，但用户还无法实际使用系统。Phase 2的目标是实现所有核心功能，让系统可以"跑通最小可玩闭环"：

1. **实现后端API**：提供创建房间、加入房间、提交消息、AI守秘人回应、掷骰、资料库管理等RESTful接口
2. **实现AI集成**：集成DeepSeek API作为叙事引擎，同时提供模拟AI用于无API key的演示
3. **实现资料库管理**：支持扫描本地资料文件夹（`sources/`）和网页上传资料文件
4. **实现前端页面和组件**：提供创建/加入房间、玩家桌面、守秘人控制台、资料库等完整用户界面

### 1.2 为什么Phase 2是"最小可玩闭环"的关键？

根据PLAN.md的定义，第一版目标是"跑通最小可玩闭环"，不是完整商业平台。这意味着：

- ✅ **必须实现**：多人房间、角色状态、资料检索、掷骰与COC核心判定、AI KP叙事
- ❌ **暂不实现**（标记为二期）：复杂私密视角、完整战斗系统、追逐系统、地图棋子、公网部署

Phase 2将聚焦于"必须实现"的部分，确保第一版可以完整演示AI守秘人体验。

## 4. 功能需求

### 4.1 后端API实现

**描述**：在 `server/app.ts` 中实现所有核心RESTful API。

#### 4.1.1 房间管理API

**端点**：
- `POST /api/rooms` - 创建房间
- `GET /api/rooms` - 列出所有房间
- `GET /api/rooms/:id` - 获取房间详情和状态

**功能需求**：
- 创建房间时，自动生成唯一房间ID
- 创建房间时，自动添加系统消息（如"房间「阿卡姆调查」已创建。"）
- 获取房间状态时，返回房间信息、参与者列表、调查员列表、最近消息

**验收标准**：
- ✅ 创建房间API返回房间对象（含ID、名称、创建时间）
- ✅ 加入房间API返回参与者对象（含ID、名称、角色）
- ✅ 获取房间状态API返回完整的房间状态JSON

#### 4.1.2 参与者管理API

**端点**：
- `POST /api/rooms/:id/join` - 加入房间

**功能需求**：
- 支持两种角色：`player`（玩家）和 `keeper`（守秘人）
- 加入房间时，自动添加系统消息（如"张三加入了调查。"）
- 加入房间后，广播房间状态更新给所有参与者

**验收标准**：
- ✅ 加入房间API正确返回参与者对象
- ✅ 加入房间后，所有参与者的房间页面实时更新

#### 4.1.3 消息管理API

**端点**：
- `POST /api/messages` - 提交消息（玩家行动、守秘人指令等）

**功能需求**：
- 支持多种消息类型：`system`（系统消息）、`player`（玩家消息）、`keeper`（守秘人消息）、`ai`（AI守秘人回应）、`roll`（掷骰结果）
- 支持多种可见性：`public`（公开）、`keeper`（仅守秘人可见）、`private`（私密，二期实现）
- 提交消息后，广播消息给房间内所有参与者（根据可见性过滤）
- 广播房间状态更新（含最新消息）

**验收标准**：
- ✅ 提交消息API正确返回消息对象
- ✅ 提交消息后，所有参与者的消息流实时更新
- ✅ `keeper` 可见性的消息仅守秘人可见

#### 4.1.4 AI守秘人回应API

**端点**：
- `POST /api/ai/respond` - 生成AI守秘人回应

**功能需求**：
- 接收玩家行动或守秘人指令作为输入
- 检索已导入资料，获取相关片段（前5个最相关结果）
- 获取房间最近12条消息作为上下文
- 组装提示词（含设定文档、资料检索结果、最近消息、当前场景、房间名称）
- 调用DeepSeek API（或模拟AI）生成结构化回应
- 解析AI回应，提取叙事文本、检定请求、状态建议、守秘人备注、引用来源
- 添加AI守秘人消息到房间消息流
- 广播消息和房间状态更新

**AI回应结构**：
```typescript
interface KeeperResponse {
  narrative: string;               // 叙事文本
  rollRequest?: RollRequest;        // 是否需要检定
  stateSuggestions: string[];      // 状态变化建议
  keeperNotes: string;             // 守秘人备注（不向玩家展示）
  sources: Array<{                 // 引用来源
    sourceName: string;
    excerpt: string;
  }>;
}
```

**验收标准**：
- ✅ AI守秘人回应API正确返回结构化回应
- ✅ 有DeepSeek API key时，调用真实API生成叙事
- ✅ 无DeepSeek API key时，使用模拟AI返回固定结构回应
- ✅ AI回应中的叙事文本、检定请求、状态建议等被正确解析和存储

#### 4.1.5 掷骰执行API

**端点**：
- `POST /api/rolls` - 执行规则检定并广播结果

**功能需求**：
- 支持多种检定类型：`skill`（技能检定）、`opposed`（对抗检定）、`combined`（组合检定）、`san`（SAN检定）
- 技能检定时，自动计算成功等级（大成功/极限成功/困难成功/常规成功/失败/大失败）
- SAN检定时，自动计算理智损失，并在损失≥5时触发INT检定和疯狂发作表
- 执行掷骰后，自动更新调查员状态（如SAN值）
- 添加掷骰结果消息到房间消息流
- 广播消息和房间状态更新

**验收标准**：
- ✅ 掷骰执行API正确返回掷骰结果
- ✅ 技能检定正确判定成功/失败和成功等级
- ✅ SAN检定正确计算理智损失，并在损失≥5时触发临时疯狂
- ✅ 掷骰结果消息正确展示在消息流中

#### 4.1.6 资料库管理API

**端点**：
- `GET /api/library` - 查看资料库状态和已导入资料片段
- `GET /api/library/search?q=<query>&limit=<limit>` - 搜索资料库
- `POST /api/library/ingest` - 扫描本地资料文件夹或处理上传文件

**功能需求**：
- 扫描本地 `sources/` 文件夹，支持 `.md`、`.txt`、`.pdf` 文件
- 上传文件时，使用Multer中间件处理 `multipart/form-data`
- PDF文件使用 `pdf-parse` 库提取文本
- 按段落切片资料文本，存入SQLite数据库
- 搜索资料时，使用简单的关键词/全文检索（第一版不用向量检索）
- 资料导入后，广播资料库更新给所有参与者

**验收标准**：
- ✅ 扫描本地资料API正确返回导入的资料片段数量
- ✅ 上传文件API正确接收和存储文件
- ✅ PDF文件正确提取文本
- ✅ 资料切片正确入库
- ✅ 搜索API正确返回相关片段

#### 4.1.7 角色卡管理API

**端点**：
- `POST /api/rooms/:id/investigators/import` - 导入 `.xlsx` 角色卡
- `PUT /api/investigators/:id` - 更新角色状态

**功能需求**：
- 导入 `.xlsx` 角色卡时，使用 `exceljs` 库解析Excel文件
- 解析角色卡中的属性、衍生值、技能、物品、伤口等
- 将解析后的角色卡存储到SQLite数据库
- 更新角色状态时，支持部分更新（如仅更新SAN值）
- 导入或更新角色卡后，广播消息和房间状态更新

**验收标准**：
- ✅ 导入角色卡API正确解析 `.xlsx` 文件
- ✅ 导入的角色卡正确存储到数据库
- ✅ 更新角色状态API正确更新指定字段
- ✅ 导入或更新后，所有参与者的界面实时更新

### 4.2 AI集成

**描述**：在 `server/ai.ts` 中实现DeepSeek API集成和模拟AI。

#### 4.2.1 DeepSeek API调用

**功能需求**：
- 使用OpenAI兼容接口（`base_url=https://api.deepseek.com`）
- 使用 `DEEPSEEK_API_KEY` 环境变量作为API密钥
- 使用 `deepseek-v4-pro` 作为默认模型（叙事任务）
- 可选使用 `deepseek-v4-flash` 作为轻量模型（简单任务）
- 注意：旧 `deepseek-chat` 模型将在2026-07-24停用，避免首版使用旧模型名

**提示词组装**：
- 系统提示词：定义AI守秘人角色、恐怖氛围编码规范、信息释放节奏控制、SAN叙事衰减指南、OC指令系统
- 资料检索结果注入：将检索到的相关片段作为上下文注入
- 最近消息历史：注入最近12条消息作为对话历史
- 当前房间状态：注入当前场景、时间线、参与者状态

**验收标准**：
- ✅ DeepSeek API正确调用，返回叙事文本
- ✅ 提示词正确组装，包含资料检索结果和消息历史
- ✅ API密钥错误时，返回友好错误信息

#### 4.2.2 模拟AI实现

**功能需求**：
- 当 `DEEPSEEK_API_KEY` 环境变量未设置时，自动使用模拟AI
- 模拟AI返回固定结构的回应，包含：
  - 叙事文本（如"你仔细搜查了图书馆，在书架后面发现了一个暗门。"）

**验收标准**：
- ✅ 无API key时，模拟AI正确返回固定结构回应
- ✅ 模拟AI回应的结构与实际AI回应的结构一致
- ✅ 使用模拟AI时，完整演示流程不被阻塞

#### 4.2.3 结构化响应解析

**功能需求**：
- 解析AI返回的叙事文本，提取叙事内容
- 解析AI返回的JSON格式检定请求（如有）
- 解析AI返回的状态建议（如有）
- 解析AI返回的守秘人备注（如有）
- 解析AI返回的引用来源（如有）

**验收标准**：
- ✅ AI响应正确解析为结构化对象
- ✅ 解析失败时有错误处理和默认值

### 4.4 前端基础实现

#### 4.4.1 主应用与路由**

**文件**：`App.tsx`、`main.tsx`

**功能需求**：
- 使用React Router实现客户端路由
- 路由定义：
  - `/` - 首页（`HomePage`）
  - `/room/:id` - 房间页面（`RoomPage`）

## 5. 非功能需求

### 5.1 性能

1. **API响应时间**：所有API端点的响应时间必须 <500ms（在不涉及AI调用的情况下）
2. **AI回应生成时间**：AI守秘人回应生成时间必须 <5秒（取决于DeepSeek API响应时间）
3. **实时通信延迟**：Socket.IO事件广播延迟必须 <100ms
4. **资料检索时间**：资料搜索API响应时间必须 <200ms

### 5.2 可用性

1. **错误提示**：所有错误必须有清晰、友好的错误提示（如API请求失败、文件上传失败、AI调用失败）
2. **加载状态**：所有长时间操作（如AI回应生成、资料扫描）必须有加载状态提示
3. **响应式设计**：所有页面和组件必须适配不同屏幕尺寸（至少支持桌面和平板）

### 5.3 可维护性

1. **代码注释**：所有复杂逻辑必须有清晰注释
2. **类型安全**：所有TypeScript代码必须启用严格模式（`strict: true`）
3. **组件复用**：通用UI模式必须抽取为可复用组件

### 5.4 兼容性

1. **浏览器兼容**：必须支持最新版本的Chrome、Firefox、Safari、Edge
2. **Node.js版本**：必须支持Node.js 18+（因为使用了 `fetch` API和 `fs/promises`）
