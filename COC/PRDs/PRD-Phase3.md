## 1. 背景与动机

### 1.1 从功能实现到质量保障

Phase 2完成了所有核心功能的实现，系统已基本可用。但在正式交付第一版之前，必须确保：

1. **功能正确性**：所有功能按预期工作，无严重Bug
2. **性能可接受**：本机多窗口演示流程顺畅，无卡顿
3. **用户体验良好**：界面美观、交互流畅、错误提示友好
4. **代码质量可靠**：测试覆盖充分，便于后续维护和扩展

Phase 3的目标就是完善测试覆盖，优化性能与用户体验，修复已知问题。

### 1.2 为什么Phase 3不可或缺？

根据PLAN.md的定义，第一版目标是"跑通最小可玩闭环"。这意味着：

- ✅ **必须确保**：核心流程（创建房间 → 加入 → 掷骰 → AI回应）100%可用
- ✅ **必须确保**：规则引擎计算100%正确（否则信任崩塌）
- 🔵 **需要优化**：UI/UX达到"可用且基本美观"的标准（不必完美，但不能有明显的别扭）
- 🔵 **需要考虑**：性能满足本机演示需求（不必优化到生产级，但演示不能卡）

Phase 3就是确保第一版达到"最小可玩闭环"的质量标准。

### 4.2 性能优化

**描述**：识别并优化性能瓶颈。

#### 4.2.1 资料检索性能优化**

**当前实现**：关键词/全文检索（简单字符串匹配）

**问题**：
- 当资料库很大时（如导入整本规则书），检索可能变慢
- 关键词匹配可能不够智能（如"侦查"匹配不到"侦察"）

**优化方案**（可选，根据性能测试结果决定）：
- **方案A**：引入向量检索（如 `sentence-transformers` 或调用Embedding API）
  - 优点：语义匹配更准确（"侦查"可以匹配"侦察"）
  - 缺点：增加系统复杂性，需要Embedding模型或API
- **方案B**：优化现有全文检索（如引入倒排索引、使用更高效的字符串匹配算法）
  - 优点：不增加系统复杂性
  - 缺点：仍然是关键词匹配，语义理解能力不足

**建议**：Phase 3先实现方案B（优化现有检索），如果资料库很大且性能确实不满意，Phase 5再考虑方案A（引入向量检索）。

**验收标准**：
- 🔵 资料检索响应时间 < 200ms（对于中等大小的资料库，如 < 1000个片段）
- 🔵 如果引入向量检索，检索响应时间 < 500ms

#### 4.2.2 前端渲染性能优化**

**当前实现**：`MessageFeed` 组件直接渲染所有消息（无虚拟滚动）

**问题**：
- 当消息很多时（如长时间游戏后），渲染可能变慢
- 滚动可能卡顿

**优化方案**：
- 引入虚拟滚动（如 `react-window` 或 `@tanstack/react-virtual`）
- 仅渲染可见区域的消息，大幅减少DOM节点数量

**验收标准**：
- 🔵 长消息流（如 > 500条消息）滚动流畅，无卡顿
- 🔵 虚拟滚动实现后，渲染性能提升 > 50%

#### 4.2.3 Socket.IO消息压缩**

**当前实现**：Socket.IO默认配置（无特殊压缩配置）

**优化方案**（可选，根据网络性能测试结果决定）：
- 配置Socket.IO消息压缩（如 `compression: true`）
- 对于大消息（如很长的AI叙事文本），考虑在发送端压缩、接收端解压

**验收标准**：
- 🔵 Socket.IO消息延迟 < 100ms（本机环境）
- 🔵 如果启用压缩，消息大小减少 > 30%

**验收标准**：

- 🔵 所有边界情况都通过手动测试或自动化测试

## 5. 非功能需求

### 5.1 性能

1. **API响应时间**：所有API端点的响应时间必须 < 500ms（在不涉及AI调用的情况下）
2. **AI回应生成时间**：AI守秘人回应生成时间必须 < 5秒（取决于DeepSeek API响应时间）
3. **实时通信延迟**：Socket.IO事件广播延迟必须 < 100ms（本机环境）
4. **资料检索时间**：资料搜索API响应时间必须 < 200ms（对于中等大小的资料库）
5. **前端渲染性能**：长消息流（> 500条）滚动帧率必须 > 30 FPS

### 5.2 可用性

1. **错误提示**：所有错误必须有清晰、友好的错误提示（如使用Alert、Toast、Inline Error Message）
2. **加载状态**：所有长时间操作必须有明确的加载状态提示（如使用Spinner、Progress Bar、Skeleton Screen）
3. **响应式设计**：所有页面和组件必须适配不同屏幕尺寸（至少支持桌面、平板、移动端）
4. **交互流畅度**：所有交互必须在 < 100ms 内有视觉反馈（如按钮点击效果、输入框即时验证）

### 5.3 可维护性

1. **测试覆盖**：单元测试覆盖率必须 > 80%，关键模块（如规则引擎、存储层）必须 > 90%
2. **代码注释**：所有复杂逻辑必须有清晰注释
3. **CI/CD**：实现自动化测试运行（如Pre-commit Hook、GitHub Actions）—— 这是可选目标，Phase 3不一定完成，可以推迟到Phase 4或Phase 5

### 5.4 兼容性

1. **浏览器兼容**：必须支持最新版本的Chrome、Firefox、Safari、Edge
2. **Node.js版本**：必须支持Node.js 18+（因为使用了 `fetch` API和 `fs/promises`）

| 移动端适配工作量超出预期 | 中 | 低 | 明确移动端适配的目标（"可用"而非"完美"）；如果工作量超出预期，可以推迟部分优化到Phase 5 |
