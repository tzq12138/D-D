## 1. 背景与动机

### 1.1 从第一版到完整系统

Phase 4完成了第一版演示与交付，系统可以"跑通最小可玩闭环"。但按照PLAN.md中的定义，第一版明确标记为"暂不做"和"二期再做"的功能包括：

1. **私密视角功能**（PLAN.md - Key Changes）
   - 暂不做复杂私密视角
   - 数据模型保留 `visibility` 字段，后续补私密线索

2. **完整战斗系统**（PLAN.md - Key Changes）
   - 战斗、追逐、完整装备价格、地图棋子先记录为二期

3. **AI一致性审查**（深度研究报告 - 幻觉抑制）
   - 一致性审查者（Consistency Reviewer）
   - 氛围审计者（Atmosphere Auditor）
   - 信息泄露哨兵（Information Leak Sentinel）

4. **素材生成**（深度研究报告 - 多模态恐怖资产生成管线）
   - 视觉资产生成（1920年代场景、NPC肖像、神话生物）
   - 听觉氛围策略（1920年代声音质感、自然与超自然的渐变）

5. **公网部署**（PLAN.md - Assumptions）
   - 第一版目标是"本机演示版网页应用"
   - 联机先做本机多窗口演示，不做公网账号、权限、部署和安全审计
   - 二期再做公网部署

Phase 5的目标就是实现这些"二期功能"，将系统从"最小可玩闭环"升级为"完整COC 7e AI KP系统"。

### 1.2 为什么Phase 5是"完整系统"的关键？

第一版的目标用户是"愿意自己动手配置DeepSeek API key的COC爱好者"。但要让系统真正走向更广阔的用户（不愿配置API key、希望开箱即用、希望公网多人游戏），必须完成Phase 5的功能：

- ✅ **第一版**：本机多窗口演示，需手动配置API key
- ⚪ **完整版（Phase 5后）**：公网多用户游戏，开箱即用（可能提供云端AI服务），支持完整COC 7e规则

## 4. 功能需求

### 4.1 战斗系统实现

**描述**：实现完整COC 7e战斗系统。

#### 4.1.1 战斗轮（Combat Round）

**功能需求**：
- 实现先攻值（Initiative）判定（DEX×5，或掷1D10 + DEX×5）
- 实现战斗轮行动顺序（按先攻值从高到低）
- 实现战斗轮行动选项：
  - 攻击（Attack）
  - 闪避（Dodge）
  - 反击（Counterattack）
  - 战技（Fighting Maneuver）
  - 瞄准（Aim）
  - 压制（Supress）
  - 装填（Reload）
  - 移动（Move）
  - 撤退（Retreat）
  - 抓捕（Grapple）
- 实现战斗轮结束条件（所有参与者完成行动，或所有敌人被击败/逃跑）

**数据模型**：
```typescript
interface CombatState {
  id: string;
  roomId: string;
  round: number;
  participants: CombatParticipant[];
  status: 'active' | 'ended';
  startedAt: string;
  endedAt?: string;
}

interface CombatParticipant {
  id: string;
  combatId: string;
  participantId: string;
  investigatorId?: string;
  initiative: number;
  actionPoints: number;
  maxActionPoints: number;
  status: 'active' | 'unconscious' | 'dead' | 'fled';
  currentAction?: string;
}
```

**验收标准**：
- ⚪ 先攻值判定正确（DEX×5，或掷1D10 + DEX×5）
- ⚪ 战斗轮行动顺序正确（按先攻值从高到低）
- ⚪ 所有战斗轮行动选项都可选择并执行
- ⚪ 战斗轮可以正确结束（所有参与者完成行动，或所有敌人被击败/逃跑）

#### 4.1.2 攻击与伤害

**功能需求**：
- 实现近战攻击检定（Fighting(Brawl) 或 Fighting(Melee Weapon) 技能）
- 实现远程攻击检定（Firearms 技能）
- 实现攻击命中判定（攻击检定结果 vs 目标闪避检定结果，或攻击检定成功且目标未闪避）
- 实现伤害计算（武器伤害骰 + 伤害加值 - 护甲减免）
- 实现伤害应用（HP当前值 -= 伤害值）
- 实现昏迷/死亡判定（HP降至0时，掷体质×5%，失败则昏迷；失败且受到伤害≥最大HP时，死亡）

**数据模型**：
```typescript
interface AttackRequest {
  id: string;
  combatId: string;
  attackerParticipantId: string;
  targetParticipantId: string;
  weaponId?: string;
  skillName: string;
  difficulty: Difficulty;
}

interface AttackResult {
  attackRoll: D100Roll;
  attackLevel: SuccessLevel;
  dodgeRoll?: D100Roll;
  dodgeLevel?: SuccessLevel;
  hit: boolean;
  damageRoll: DiceExpression;
  damageTotal: number;
  armorReduction: number;
  finalDamage: number;
  targetNewHp: number;
  targetStatus: 'active' | 'unconscious' | 'dead';
}
```

**验收标准**：
- ⚪ 攻击检定正确执行（近战/远程，考虑技能值和难度）
- ⚪ 命中判定正确（考虑目标是否闪避）
- ⚪ 伤害计算正确（武器伤害骰 + 伤害加值 - 护甲减免）
- ⚪ 伤害正确应用（HP当前值减少）
- ⚪ 昏迷/死亡判定正确（HP降至0时）

#### 4.1.3 闪避与反击

**功能需求**：
- 实现闪避检定（Dodge 技能）
- 实现闪避成功效果（避免伤害）
- 实现闪避失败效果（承受全额伤害）
- 实现反击检定（Fighting(Brawl) 或 Fighting(Melee Weapon) 技能）
- 实现反击成功效果（对攻击者造成伤害）
- 实现反击失败效果（无额外效果）

**验收标准**：
- ⚪ 闪避检定正确执行（考虑Dodge技能值）
- ⚪ 闪避成功/失败效果正确应用
- ⚪ 反击检定正确执行（考虑技能值和难度）
- ⚪ 反击成功/失败效果正确应用

#### 4.1.4 重伤（Major Wound）

**功能需求**：
- 实现重伤判定（单次受到伤害 ≥ 最大HP的一半）
- 实现重伤效果（
  - 昏迷判定（掷体质×5%，失败则昏迷）
  - 大出血判定（每轮损失1D6 HP，直到接受治疗或死亡）
  - 永久伤残判定（昏迷轮数×5%的概率，失败则永久失去1D10×5% CON或STR）
- 实现重伤治疗（First Aid 或 Medicine 技能）

**验收标准**：
- ⚪ 重伤判定正确（单次受到伤害 ≥ 最大HP的一半）
- ⚪ 重伤效果正确应用（昏迷、大出血、永久伤残）
- ⚪ 重伤治疗正确执行（First Aid 或 Medicine 技能检定）

#### 4.1.5 移动与追击

**功能需求**：
- 实现移动动作（消耗1个动作点，移动距离 = MOV×2码）
- 实现冲锋动作（消耗2个动作点，移动距离 = MOV×4码，攻击检定承受一个惩罚骰）
- 实现追击判定（追逐者移动距离 > 逃跑者移动距离时，追击成功）
- 实现逃脱判定（逃跑者移动距离 > 追逐者移动距离时，逃脱成功）

**验收标准**：
- ⚪ 移动/冲锋动作正确执行（消耗正确动作点，移动正确距离）
- ⚪ 追击/逃脱判定正确（比较移动距离）

### 4.2 追逐系统实现

**描述**：实现完整COC 7e追逐系统。

#### 4.2.1 追逐轮（Chase Round）

**功能需求**：
- 实现追逐轮（Chase Round）概念（1轮 = 1D6+MOV秒）
- 实现追逐参与者（Runner 和 Pursuer）
- 实现追逐轮行动顺序（所有参与者同时声明动作）
- 实现追逐轮行动选项：
  - 全速移动（All-out Movement）：移动距离 = MOV×2D6码，但承受一个惩罚骰于下一个敏捷或体质检定
  - 正常移动（Normal Movement）：移动距离 = MOV×1D6码
  - 小心移动（Careful Movement）：移动距离 = MOV×1D6/2码，但获得一个奖励骰于下一个敏捷或体质检定
  - 设置障碍（Set Hazard）：消耗1个动作，放置一个障碍
  - 跨越障碍（Cross Hazard）：移动距离 -= 障碍的DC，或检定失败则停止移动
- 实现追逐轮结束条件（追逐者捕获逃跑者，或逃跑者逃脱）

**数据模型**：
```typescript
interface ChaseState {
  id: string;
  roomId: string;
  round: number;
  participants: ChaseParticipant[];
  hazards: Hazard[];
  status: 'active' | 'ended';
  result?: 'captured' | 'escaped';
  startedAt: string;
  endedAt?: string;
}

interface ChaseParticipant {
  id: string;
  chaseId: string;
  participantId: string;
  investigatorId?: string;
  role: 'runner' | 'pursuer';
  moveRate: number;
  actionPoints: number;
  maxActionPoints: number;
  currentPosition: number;
  status: 'active' | 'stopped' | 'captured';
}

interface Hazard {
  id: string;
  chaseId: string;
  name: string;
  position: number;
  dc: number;
  setterParticipantId: string;
}
```

**验收标准**：
- ⚪ 追逐轮正确初始化（所有参与者、回合数）
- ⚪ 追逐轮行动顺序正确（所有参与者同时声明动作）
- ⚪ 所有追逐轮行动选项都可选择并执行
- ⚪ 追逐轮可以正确结束（追逐者捕获逃跑者，或逃跑者逃脱）

#### 4.2.2 障碍（Hazard）系统**

**功能需求**：
- 实现障碍放置（消耗1个动作，设置障碍的位置和DC）
- 实现障碍跨越（移动距离 -= 障碍的DC，或检定失败则停止移动）
- 实现障碍类型：
  - 物理障碍（如锁门、倒塌的书架）：需要 Strength 或 Locksmith 检定
  - 感知障碍（如浓雾、黑暗）：需要 Spot Hidden 或 Listen 检定
  - 社交障碍（如拥挤的市场、盘问的警卫）：需要 Fast Talk 或 Stealth 检定
- 实现障碍持续时间（一次性，或持续N轮）

**验收标准**：
- ⚪ 障碍正确放置（位置、DC、类型）
- ⚪ 障碍正确跨越（移动距离减少或检定）
- ⚪ 所有障碍类型都可正确处理

#### 4.2.3 追逐胜负判定**

**功能需求**：
- 实现捕获条件（追逐者与逃跑者的位置相同，且追逐者拥有剩余动作点）
- 实现逃脱条件（逃跑者的位置 ≥ 逃脱距离（通常为起点 + MOV×10码），或追逐者停止追逐）
- 实现追逐结果叙事化（根据捕获/逃脱，生成对应的叙事文本）

**验收标准**：
- ⚪ 捕获条件正确判定
- ⚪ 逃脱条件正确判定
- ⚪ 追逐结果正确叙事化（AI生成或手动编写）

### 4.3 私密线索与信息控制实现

**描述**：实现按 `visibility` 过滤消息，实现私密线索展示。

#### 4.3.1 按 `visibility` 过滤消息**

**功能需求**：
- 在前端消息流组件（`MessageFeed.tsx`）中，根据当前用户的角色和 `visibility` 字段过滤消息
- `visibility = 'public'`：所有参与者可见
- `visibility = 'keeper'`：仅守秘人可见
- `visibility = 'private'`：仅特定调查员可见（需要 `metadata.visibleToParticipantIds` 字段）

**数据模型变更**：
```typescript
interface Message {
  // ... 其他字段
  visibility: Visibility;
  metadata?: {
    visibleToParticipantIds?: string[]; // 仅当 visibility = 'private' 时使用
    // ... 其他元数据
  };
}
```

**验收标准**：
- ⚪ `visibility = 'public'` 的消息对所有参与者可见
- ⚪ `visibility = 'keeper'` 的消息仅对守秘人可见
- ⚪ `visibility = 'private'` 的消息仅对 `metadata.visibleToParticipantIds` 中列出的参与者可见

#### 4.3.2 私密线索展示**

**功能需求**：
- 在守秘人控制台中，提供"添加私密线索"按钮
- 点击后，弹出表单：
  - 线索描述（仅守秘人可见）
  - 线索描述（玩家可见版本）
  - 可见范围（选择哪些调查员可以看到此线索）
- 提交后，创建两条消息：
  - 一条 `visibility = 'keeper'`，内容为线索描述（仅守秘人可见）
  - 一条 `visibility = 'private'`，内容为线索描述（玩家可见版本），`metadata.visibleToParticipantIds` 设置为选择的调查员
- 在玩家界面，仅当自己的参与者ID在 `visibleToParticipantIds` 中时，才显示此消息

**验收标准**：
- ⚪ 守秘人可以成功添加私密线索
- ⚪ 私密线索正确存储为两条消息（一条守秘人可见，一条玩家可见版本）
- ⚪ 玩家只能看到自己参与的私密线索

#### 4.3.3 守秘人视角切换**

**功能需求**：
- 在守秘人控制台中，提供"视角切换"下拉菜单
- 选项包括：
  - "守秘人视角"（默认，看到所有消息）
  - "作为 [调查员名称] 视角"（仅看到该调查员可见的消息）
- 切换后，消息流组件（`MessageFeed.tsx`）根据选择的视角过滤消息

**验收标准**：
- ⚪ 守秘人可以成功切换视角
- ⚪ 切换视角后，消息流正确过滤（仅显示当前视角可见的消息）

### 4.4 完整装备价格系统实现

### 4.6 AI一致性审查实现

**描述**：实现线索一致性验证器、氛围审计器、信息泄露哨兵。

#### 4.6.1 线索一致性验证器（Consistency Reviewer）**

**功能需求**：
- 在AI生成叙事回应后，调用一致性验证器
- 一致性验证器使用另一个LLM（temperature=0，如GPT-4o）检查AI叙事是否与已有线索图谱中的任何事实相矛盾
- 如果发现矛盾，暂停回应并提示AI修正
- 如果没有矛盾，允许回应通过

**技术实现**：
- 创建 `server/consistencyReviewer.ts` 模块
- 导出 `reviewNarrative(roomId: string, narrative: string): Promise<ReviewResult>` 函数
- `ReviewResult` 接口：
  ```typescript
  interface ReviewResult {
    passed: boolean;
    contradictions?: string[];
    suggestions?: string[];
  }
  ```

**验收标准**：
- ⚪ 一致性验证器正确检测AI叙事与线索图谱的矛盾
- ⚪ 发现矛盾时，正确暂停回应并提示AI修正
- ⚪ 无矛盾时，正确允许回应通过

#### 4.6.2 氛围审计器（Atmosphere Auditor）**

**功能需求**：
- 在AI生成叙事回应后，调用氛围审计器
- 氛围审计器使用另一个LLM（被提示为"洛夫克拉夫特文学评论家"，如Claude）审查当前回应的恐怖质量
- 标记滥用的血腥描述、现代俚语和过于阳光的叙事语调
- 如果氛围质量不达标，提示AI修正

**技术实现**：
- 创建 `server/atmosphereAuditor.ts` 模块
- 导出 `auditNarrative(narrative: string, currentTension: number): Promise<AuditResult>` 函数
- `AuditResult` 接口：
  ```typescript
  interface AuditResult {
    passed: boolean;
    issues?: string[];
    suggestions?: string[];
  }
  ```

**验收标准**：
- ⚪ 氛围审计器正确审查AI叙事的恐怖质量
- ⚪ 正确标记滥用的血腥描述、现代俚语和过于阳光的叙事语调
- ⚪ 氛围质量不达标时，正确提示AI修正

#### 4.6.3 信息泄露哨兵（Information Leak Sentinel）**

**功能需求**：
- 在AI生成叙事回应后，调用信息泄露哨兵
- 信息泄露哨兵检查回应中是否包含当前调查员不应知道的信息
- 检查依据：`investigator_knowledge_state` 块（哪些线索已知、哪些线索未知）
- 如果发现信息泄露，暂停回应并提示AI修正

**技术实现**：
- 创建 `server/informationLeakSentinel.ts` 模块
- 导出 `checkLeak(narrative: string, roomId: string, participantId: string): Promise<LeakCheckResult>` 函数
- `LeakCheckResult` 接口：
  ```typescript
  interface LeakCheckResult {
    passed: boolean;
    leaks?: string[];
    suggestions?: string[];
  }
  ```

**验收标准**：
- ⚪ 信息泄露哨兵正确检查回应中是否包含不应知道的信息
- ⚪ 发现信息泄露时，正确暂停回应并提示AI修正
- ⚪ 无泄露时，正确允许回应通过
